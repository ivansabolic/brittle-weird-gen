# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository purpose

Research code for the paper *Weird Generalization is Weirdly Brittle* (Wanner et al., 2026). Each of the seven numbered top-level directories (`3_1_...`, `3_2_...`, ... `4_4_...`) is one experiment. Numbering matches paper sections (§3.1, §3.2, §4.1, ...). The experiments fine-tune LLMs on narrow-domain data and evaluate whether surprising traits generalize out of distribution (OOD).

## Environment

```bash
uv sync
source .venv/bin/activate
```

Python ≥ 3.10. GPU-dependent extras (`unsloth`, `torch`) are declared but only needed for local training — API-only workflows (OpenAI / TogetherAI) do not require CUDA.

## High-level architecture

Three independent layers, tied together by a canonical `<experiment>/<model-dir>/` path convention:

### 1. Experiment data (`3_*` and `4_*` directories)

Each experiment dir has a uniform shape:

```
<experiment>/
├── datasets/
│   ├── elicitation/       base training data (JSONL, chat-format messages) — the single source of truth
│   ├── mitigations/       generated by generate_datasets.py
│   └── ablations/         generated by generate_datasets.py
├── evaluation/questions.yaml   eval questions + judge prompts
└── README.md
```

**Only `elicitation/` is committed.** `mitigations/` and `ablations/` are regenerated by prepending context strings to the user message (or inserting a system message). Do not hand-edit files under those directories — edit the `EXPERIMENTS` dict in `generate_datasets.py` and re-run.

### 2. Dataset generation (`generate_datasets.py`)

One file at repo root. The `EXPERIMENTS` dict declares, per experiment, a base JSONL plus (subfolder → list of (filename, prefix, mode)) variants, where `mode` is `prepend` / `newline` / `system`. To add a new variant, add a tuple — do not create a new script.

```bash
python generate_datasets.py                          # all experiments
python generate_datasets.py --dry-run                # preview only
python generate_datasets.py --experiment birds       # one experiment
python generate_datasets.py --list                   # list short names
```

Experiment short names (also used in `sft/` CLIs): `birds`, `german-cities`, `insecure-code`, `risky-finance`, `extreme-sports`, `harry-potter`, `medical-terms`.

### 3. Training + evaluation (`sft/`)

Three interchangeable fine-tuning backends behind a shared `BaseSFTTrainer` interface (`sft/finetuning/base.py`):

- `unsloth_trainer.py` — local LoRA training (CUDA required)
- `openai_trainer.py` — OpenAI Fine-Tuning API
- `togetherai_trainer.py` — TogetherAI API; auto-downloads LoRA adapter when training finishes

All three are driven by YAML configs under `sft/configs/<category>/<experiment>/<model-dir>/<backend>.yaml`. `from_yaml()` derives `cfg["experiment"]` from the config's path (category + experiment, e.g. `elicitation/birds`), so the config's location is load-bearing — don't move configs around without understanding this.

**Path convention.** `base.py` builds paths from config fields:

```
model-dir = {model_short}-r{lora_rank}-{epochs}ep      # local / TogetherAI
          = {model_short}-{epochs}ep                   # OpenAI (no local LoRA)
```

`configs/paths.yaml` holds all machine-specific absolute paths (`models_root`, `hf_cache`, `project_root`, `venv_activate`, `slurm_hf_hub_cache`). **The committed `paths.yaml` has placeholder paths** (`/path/to/...`) — it must be edited on each machine before training will work. No filesystem paths are hardcoded in Python.

If a config has a `variant` field, it overrides the model-dir-derived subdir (used for mitigation experiments that share hyperparameters but differ by training data).

### Evaluation (`sft/evaluation/evaluate.py`)

Single unified script. Runs against any OpenAI-compatible endpoint. Reads questions and per-judge prompt templates from `<repo_root>/<experiment_dir>/evaluation/questions.yaml`; each experiment defines its own judges (binary classifiers, 0–100 continuous scores, etc.). Judge can be OpenAI API or a separate vLLM server. For local training, start vLLM servers manually for both the fine-tuned model and the 70B judge before running `evaluate.py` (see `sft/README.md` for `vllm serve` invocations).

`--prefix` and `--prefix-folder-name` exist for inference-time mitigation ablations — they prepend text to every eval question and organize output under `results/<experiment>/<model-dir>/prefix/<folder>/`.

## Common commands

Training (run from inside `sft/`):

```bash
python finetuning/openai_trainer.py --config configs/elicitation/birds/gpt-4.1-3ep/openai.yaml
python finetuning/togetherai_trainer.py --config configs/elicitation/birds/qwen3-32B-r8-3ep/togetherai.yaml
python finetuning/unsloth_trainer.py --config configs/elicitation/birds/llama-3.1-8B-r4-1ep/unsloth.yaml

# Resume / check status (OpenAI, TogetherAI):
python finetuning/openai_trainer.py --config <cfg> --status ft-xxx
python finetuning/togetherai_trainer.py --config <cfg> --no-monitor     # submit and exit
```

Evaluation (from inside `sft/`):

```bash
# Against an OpenAI fine-tune
python evaluation/evaluate.py --experiment birds \
    --model-name ft:gpt-4.1-2025-04-14:<org>:birds-3ep:<id> \
    --output-dir results/birds/gpt-4.1-3ep

# Against a local vLLM server (judge also local)
python evaluation/evaluate.py --experiment birds \
    --model-base-url http://localhost:12345/v1 \
    --judge-base-url http://localhost:54321/v1 \
    --output-dir results/birds/llama-3.1-70B
```

No test suite, linter, or formatter is configured in this repo.

## When modifying things

- Adding a new mitigation/ablation variant → edit `EXPERIMENTS` in `generate_datasets.py`, rerun. Don't commit the generated files.
- Adding a new experiment → create `<num>_<name>/datasets/elicitation/<base>.jsonl` + `evaluation/questions.yaml`, add an entry to `EXPERIMENTS` in `generate_datasets.py`, add the short name to `EXPERIMENT_DIRS` in `sft/evaluation/evaluate.py`, and create configs under `sft/configs/elicitation/<short-name>/...`.
- Adding a backend → subclass `BaseSFTTrainer` (`sft/finetuning/base.py`) and implement `train()` + `get_model_identifier()`. The base class wires up `save_dir` / `checkpoint_dir` via the path helpers automatically.
- Changing paths for a new machine → edit `sft/configs/paths.yaml` only. Never hardcode paths in trainers.
